## Phase 4：同步层 + 清理

> ✅ **已完成（2026-07-09）**

### 目标

实现本地 libSQL ↔ Turso 云端的 Embedded Replica 双向同步，清理旧 NocoBase 手写同步架构。

历史阶段 4.1–4.5 曾基于 NocoBase 手写 REST 同步实现完整的推送/拉取/冲突处理，此后已整体废弃并迁移至 Turso Embedded Replica。以下 4.6 记录该迁移的四个阶段执行情况。

### 4.6 同步后端迁移：NocoBase → Turso（libSQL）

> 目标：用 Turso Embedded Replica 替换 NocoBase 手写同步。保留本地优先体验，删除 `retry_count / synced / last_error / nocobase_id / nocobase_updated_at / local_updated_at` 等 NocoBase 遗留列在业务代码中的使用（表结构暂保留列作为历史遗留，后由 §4.7 彻底清理），删除 `db/nocobase/`、`commands/sync.rs`、`db/sync_log.rs`、`db/preferences.rs`。

- [x] **阶段 1 · 数据搬运脚本**：[`scripts/migrate-to-turso/`](../scripts/migrate-to-turso/) 独立 Node 脚本（better-sqlite3 + @libsql/client），支持 `--dry-run / --drop / --verify`；不动应用代码，仅把本地 accounting.db 全量搬到 Turso 并顺便剔除 NocoBase 列
- [x] **阶段 2 · Rust 侧驱动切换**：rusqlite → libsql 0.9（异步 API）；`AppState` 用 `tauri::async_runtime::block_on` 初始化；Tauri commands 改 async；`Database` 内部持有 `Arc<libsql::Database>`，提供 `new()` 本地模式与 `new_with_turso()` Embedded Replica 模式；时间戳生成从 SQL `datetime('now','localtime')` 迁移到 Rust `chrono::Local` 参数传入
- [x] **阶段 3 · 清理 NocoBase**：删除 `src-tauri/src/db/nocobase/`、`commands/sync.rs`、`db/sync_log.rs`、`db/preferences.rs`；前端 `src/api/tauri.ts` 删除 `syncFull / syncPush / syncPull / getSyncLogs`；前端类型删除 `synced / retry_count / last_error / local_updated_at` 与两个 NocoBase 配置项；`AGENTS.md` §4.2 重写为 Turso 同步规范；`CODE_WIKI.md` §9 重写为 Turso 同步机制
- [x] **阶段 4 · 新 Settings 页**：Turso 启用开关 + URL / Token 输入 + 测试连接（`test_turso_connection`）+ 立即同步一次（`sync_turso`）；新增 3 个 `app_config` key：`turso_sync_enabled` / `turso_url` / `turso_token`

### 4.8 启动同步策略调整：延迟静默同步（2026-07-09）

> 目标：解决 Embedded Replica 启动同步与 schema init + 前端并发查询之间的 `database is locked` 竞争；同时让应用启动秒开、契合本地优先设计。

**问题**：`Database::new_with_turso()` 内部先跑 `db.sync()`（阻塞首次同步）再 `schema::init()`；应用启动后前端 Home / Records / Stats 多个 store 同时挂载并发 invoke，与 libsql 后台 sync worker 竞争 WAL 锁 → 首页偶发"加载失败：Failed to get connection: database is locked"。

- [x] `connection.rs::new_with_turso` 删除启动阶段的 `db.sync().await`；只 open replica，让 schema init 立刻在本地 replica 上运行
- [x] `get_conn()` 改为 async；每次新连接加 `PRAGMA busy_timeout=5000` + 连接失败时的 lock 重试（最多 15 × 200ms = 3s）
- [x] Home.vue 抽出 `loadData()`；`onMounted` 先 `loadData()` 秒开界面，再调用 `scheduleBackgroundSync()`——3s 后静默调用 `sync_turso`，同步成功后 `loadData()` 静默刷新数据 + toast 提示（"云端已同步" / "云端已同步（数据已更新）"）
- [x] 同步失败静默忽略（未启用 Turso / 网络问题都不打扰用户），本地数据继续可用
- [x] 文档同步：`AGENTS.md` §4.2 / §7 更新触发时机描述；`CODE_WIKI.md` §9.1 补充三种触发方式；附录 C 更新潜在改进点

### 4.9 平台相关配置隔离：active_python_path 按 OS 分 key（2026-07-09）

> 目标：Turso 双向同步 `app_config` 时，避免 macOS 与 Windows 互相覆盖 Python 路径。

- [x] `commands/ocr.rs`：新增 `PY_KEY` 常量（`cfg` 编译期选择 `_macos` / `_windows` / `_linux` 后缀），`get_active_python_path` / `set_active_python_path` 全部走此常量
- [x] `commands/config.rs::AppConfig::new`：启动时若发现旧 key `active_python_path`，自动迁移到当前平台的新 key（新 key 已存在时不覆盖），随后删除旧 key
- [x] `CODE_WIKI.md` §7.1 表格 / §7.2.OCR 描述、`AGENTS.md` §7 避坑清单 同步
