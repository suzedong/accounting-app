## 规则

1. **文档先行**：任何需求变更、架构调整、界面改动或代码修改，都需先讨论确认，然后同步更新相关文档。
2. **架构变更需讨论**：永远不要在未和用户讨论并确认的情况下擅自改动项目架构。

## Why

- 用户明确要求文档先行，避免代码和文档脱节导致后期维护混乱
- 架构变更可能带来意外的副作用，需要用户主动评估风险

## How to apply

- **需求/架构变更** → 先讨论 → 确认后更新 `doc/01-project-overview.md`
- **界面/交互变更** → 先讨论 → 确认后更新 `doc/01-project-overview.md`（界面设计章节）
- **开发计划调整** → 先讨论 → 确认后更新 `doc/02-development-roadmap.md`
- **重构方案变化** → 先讨论 → 确认后更新 `doc/03-active-design-docs.md`
- **代码编写/修改** → 在文档已更新的基础上进行
- **CLAUDE.md** → 涉及架构/数据模型/目录结构变化时同步更新
- **涉及架构调整时**（删除/新增模块、重构目录结构、替换技术方案等），先停下来提供修改方案（为什么改、怎么改、影响范围），等待用户确认
- 小范围的 bug 修复、代码补全、UI 微调不需要确认
